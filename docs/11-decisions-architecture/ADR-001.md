# ADR-001 : Utiliser RabbitMQ comme broker de messages pour la communication inter-modules

| Champ | Valeur |
|-------|--------|
| **N° ADR** | 001 |
| **Titre** | Utiliser RabbitMQ comme broker de messages pour la communication inter-modules |
| **Date** | 2026-05-13 |
| **Statut** | Accepté |
| **Auteur(s)** | Adrien DETEE |

## Contexte

SupEvents est architecturé en modules distincts côté backend (NestJS) : `TicketModule`, `PaymentModule`, `NotificationModule`, `EventModule`. Ces modules doivent communiquer lors d'actions critiques — notamment, la confirmation d'un ticket après capture du paiement Stripe doit déclencher l'envoi d'une notification email.

Une communication synchrone directe entre modules (`TicketModule` appelle `NotificationModule` via HTTP) créerait un couplage fort : une indisponibilité de SendGrid bloquerait la confirmation des tickets. La charge attendue lors des événements populaires (500 inscriptions simultanées) exige que le flux critique d'inscription ne soit jamais bloqué par des services non-critiques.

## Décision

Nous avons décidé d'adopter **RabbitMQ** comme broker de messages central pour la communication asynchrone entre modules. Les modules producteurs (`TicketModule`, `PaymentModule`) publient des événements métier dans un exchange de type `topic`. Les modules consommateurs (`NotificationWorker`) s'abonnent aux queues correspondantes via des binding keys.

## Alternatives envisagées

| Option | Avantages | Inconvénients | Raison d'exclusion |
|--------|-----------|---------------|-------------------|
| **Apache Kafka** | Très haute performance, rétention longue, replay | Complexité opérationnelle élevée, nécessite ZooKeeper ou KRaft, expertise requise | Surdimensionné pour le volume SupEvents (~500 events/h peak) ; équipe sans expérience Kafka |
| **Redis Pub/Sub** | Légèreté, faible latence, Redis déjà présent | Messages non persistants (perte en cas de redémarrage), pas de garantie de livraison, pas d'ACK | Inacceptable pour des notifications critiques (confirmation de ticket) |
| **Appels HTTP synchrones** | Simplicité d'implémentation | Couplage fort, cascade d'échecs, latence ajoutée sur le flux principal | Contraire à l'objectif de découplage |
| **RabbitMQ (retenu)** | Mature, queues persistantes, ACK/NACK, DLQ, Docker officiel, interface admin | Performance moindre que Kafka, pas adapté au streaming analytique | N/A — option retenue |

## Conséquences

### Positives

- Découplage effectif : une panne de `NotificationWorker` ou de SendGrid n'impacte pas les inscriptions.
- Garantie de livraison des messages grâce aux ACK et aux queues durables.
- Dead Letter Queue (DLQ) pour inspecter et rejouer les notifications échouées.
- Ajout futur de nouveaux consommateurs sans modifier les producteurs.

### Négatives

- Ajout d'une dépendance d'infrastructure supplémentaire à maintenir (RabbitMQ container).
- Les développeurs doivent maîtriser les concepts AMQP (exchange, binding, routing key).
- L'ordre de livraison des messages n'est pas garanti avec plusieurs consommateurs en parallèle.

### Dette technique acceptée

La configuration avancée des DLQ (retry avec backoff exponentiel, alertes automatiques après N échecs en DLQ) est reportée en phase 2. En phase 1, les messages en DLQ sont tracés en base et nécessitent une intervention manuelle pour le re-traitement.
