# ADR-002 : Utiliser des verrous pessimistes PostgreSQL pour la gestion concurrentielle des places

| Champ | Valeur |
|-------|--------|
| **N° ADR** | 002 |
| **Titre** | Utiliser des verrous pessimistes PostgreSQL pour la gestion concurrentielle des places |
| **Date** | 2026-05-13 |
| **Statut** | Accepté |
| **Auteur(s)** | Adrien DETEE |

## Contexte

SupEvents doit garantir qu'un événement ne soit jamais en surbooking, même lors de pics de 500 inscriptions simultanées. Le scénario critique est la dernière place disponible : deux étudiants cliquent sur "S'inscrire" exactement au même moment. Sans mécanisme de contrôle, les deux réservations pourraient être acceptées.

Le système utilise PostgreSQL comme base de données principale. La contrainte de charge (500 req/s peak) est significative mais pas extrême — l'approche doit être correcte en premier lieu, performante en second.

## Décision

Nous avons décidé d'utiliser des **verrous pessimistes PostgreSQL** via `SELECT ... FOR UPDATE` sur la ligne de l'événement lors de la réservation. La transaction de réservation lit le nombre de places disponibles avec le verrou, décrémente, puis valide ou annule selon la disponibilité.

```sql
BEGIN;
SELECT available_places FROM events WHERE id = $1 FOR UPDATE;
-- Si available_places > 0 : décrémenter et créer le ticket
-- Sinon : rollback et retourner 409
COMMIT;
```

## Alternatives envisagées

| Option | Avantages | Inconvénients | Raison d'exclusion |
|--------|-----------|---------------|-------------------|
| **Verrou optimiste (version field)** | Meilleure performance en cas de contention faible | Nécessite des retries côté application, complexité accrue, risque de boucles infinies sous forte charge | Sous forte contention (500 req simultanées sur le même événement), le taux d'échec et de retry serait trop élevé |
| **Redis DECR atomique** | Très performant, atomique par nature | Double source de vérité (Redis + PostgreSQL), synchronisation complexe, risque de désync en cas de crash Redis | Complexité opérationnelle disproportionnée pour la taille du projet |
| **Queue de réservation (FIFO)** | Garantie absolue de l'ordre | Latence artificielle inacceptable, complexité architecturale | Surdimensionné pour 500 req/s |
| **SELECT FOR UPDATE (retenu)** | Garantie forte, simple, natif PostgreSQL | Contention sur la ligne d'événement en pic | N/A — option retenue |

## Conséquences

### Positives

- Garantie absolue d'absence de surbooking — PostgreSQL sérialise les transactions concurrentes sur la même ligne.
- Cohérence forte sans logique de retry côté application.
- Comportement prévisible et simple à débugger.

### Négatives

- Contention élevée sur la ligne de l'événement lors des pics de demande (500 req/s). Les transactions s'exécutent en série, non en parallèle, sur cet événement.
- Temps de verrou moyen estimé à 5-10ms par transaction — acceptable pour notre SLO de 500ms p95.

### Dette technique acceptée

Si la charge dépasse 1 000 req/s sur un même événement, envisager un système de queue de réservation asynchrone ou un sharding de la disponibilité. Ce scénario est hors périmètre phase 1.
