# ADR-003 : Déléguer l'authentification au SSO OIDC de l'école plutôt qu'une gestion d'identités locale

| Champ | Valeur |
|-------|--------|
| **N° ADR** | 003 |
| **Titre** | Déléguer l'authentification au SSO OIDC de l'école plutôt qu'une gestion d'identités locale |
| **Date** | 2026-05-13 |
| **Statut** | Accepté |
| **Auteur(s)** | Adrien DETEE |

## Contexte

SupEvents doit être accessible exclusivement aux membres de l'établissement (étudiants et personnel). L'école dispose d'un SSO institutionnel compatible OIDC/OAuth2. L'alternative serait de créer un système d'authentification propre à SupEvents (inscription par email, gestion des mots de passe, récupération de compte).

Les contraintes clés sont : (1) ne pas dupliquer les identités institutionnelles, (2) limiter la surface d'attaque sur la gestion des credentials, (3) simplifier l'expérience utilisateur (un seul compte pour accéder à tous les services de l'école).

## Décision

Nous avons décidé de **déléguer entièrement l'authentification au SSO OIDC de l'école**. SupEvents implémente uniquement le rôle de Relying Party OIDC : il redirige l'utilisateur non authentifié vers le SSO, reçoit le code d'autorisation via callback, échange le code contre un ID token, valide la signature du token, et émet un JWT applicatif propre à SupEvents pour les appels API internes.

SupEvents ne stocke pas de mots de passe. La table `user` ne contient que les métadonnées de profil (email, prénom, rôle RBAC) synchronisées depuis le SSO lors du premier login.

## Alternatives envisagées

| Option | Avantages | Inconvénients | Raison d'exclusion |
|--------|-----------|---------------|-------------------|
| **Auth locale (email + password)** | Indépendance du SSO, gestion fine des politiques de mot de passe | Gestion des mots de passe (hashage, reset, rotation), surface d'attaque supplémentaire, doublon d'identités | Complexité sécurité disproportionnée, mauvaise UX (compte supplémentaire) |
| **Auth Google OAuth** | Simple à implémenter | Exclut les membres sans compte Google, ne reflète pas l'appartenance institutionnelle | Ne garantit pas l'appartenance à l'établissement |
| **SSO OIDC École (retenu)** | Identités institutionnelles, UX unifiée, sécurité gérée par l'école | Dépendance à la disponibilité du SSO, limiter les cas de panne SSO | N/A — option retenue |

## Conséquences

### Positives

- Aucune gestion de mots de passe dans SupEvents — surface d'attaque réduite.
- UX simplifiée : les étudiants utilisent leurs identifiants institutionnels habituels.
- Garantie que seuls les membres de l'établissement accèdent à la plateforme.
- Conformité RGPD facilitée : les données d'identité sont sous la responsabilité de l'école.

### Négatives

- Dépendance à la disponibilité du SSO de l'école : si le SSO est indisponible, les utilisateurs non connectés ne peuvent pas se connecter.
- Les sessions JWT applicatifs (Redis) ont une durée de vie de 1h — à l'expiration, un refresh via le SSO est nécessaire si le refresh token est expiré.
- Intégration technique spécifique au SSO de l'école (URL d'issuer, scopes disponibles) à documenter et tester.

### Dette technique acceptée

La gestion du cas de panne du SSO (mode dégradé avec sessions prolongées pour les utilisateurs déjà connectés) est reportée en phase 2. En phase 1, une indisponibilité du SSO empêche toute nouvelle connexion.
