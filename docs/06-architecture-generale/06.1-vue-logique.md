# §6.1 — Vue logique

## 6.1.1 Diagramme C4 Context

Ce diagramme présente la vue grand-angle du système SupEvents, tel qu'il est perçu par ses utilisateurs et les systèmes tiers avec lesquels il interagit. Il est destiné aux décideurs métier et aux nouveaux membres de l'équipe souhaitant comprendre le positionnement du système sans entrer dans les détails techniques.

```mermaid
flowchart TD
    etudiant(["👤 Étudiant\n[Personne]\nConsulte et s'inscrit\naux événements"])
    organisateur(["👤 Organisateur\n[Personne]\nCrée et gère\nles événements"])
    admin(["👤 Administrateur\n[Personne]\nValide les organisateurs\net supervise la plateforme"])

    supevents["🖥️ SupEvents\n[Système logiciel]\nPlateforme de gestion\nd'événements étudiants"]

    stripe["💳 Stripe\n[Système externe]\nPaiement en ligne"]
    sendgrid["📧 SendGrid\n[Système externe]\nEmail transactionnel"]
    sso["🔐 SSO École\n[Système externe]\nAuthentification OIDC"]
    cdn["🌐 CDN\n[Système externe]\nDiffusion des médias\net assets statiques"]

    etudiant -- "HTTPS/REST\nConsulte, s'inscrit, paie" --> supevents
    organisateur -- "HTTPS/REST\nCrée et gère les événements" --> supevents
    admin -- "HTTPS/REST\nAdministre la plateforme" --> supevents

    supevents -- "HTTPS/REST\nInitie les paiements\n(PaymentIntent)" --> stripe
    stripe -- "Webhook HTTPS\nConfirme/refuse\nla transaction" --> supevents
    supevents -- "HTTPS/REST\nEnvoie les emails\ntransactionnels" --> sendgrid
    supevents -- "OIDC/OAuth2\nDélègue l'authentification" --> sso
    sso -- "OIDC Callback\nJWT d'identité" --> supevents
    supevents -- "HTTPS\nSert les médias\net assets" --> cdn

    style supevents fill:#1168bd,color:#fff
    style etudiant fill:#08427b,color:#fff
    style organisateur fill:#08427b,color:#fff
    style admin fill:#08427b,color:#fff
    style stripe fill:#666,color:#fff
    style sendgrid fill:#666,color:#fff
    style sso fill:#666,color:#fff
    style cdn fill:#666,color:#fff
```

SupEvents est le point de coordination central : il ne gère pas les identités (délégué au SSO OIDC de l'école) ni le transport des paiements (délégué à Stripe). Ce choix de délégation permet de rester hors du périmètre PCI-DSS et de réutiliser les comptes institutionnels existants des étudiants. Le CDN sert les assets statiques du frontend, déchargeant l'API backend des requêtes de ressources statiques.

## 6.1.2 Diagramme C4 Containers

Ce diagramme zoome à l'intérieur de SupEvents pour montrer les blocs déployables qui le composent. Il est destiné aux développeurs et architectes qui doivent comprendre la décomposition technique du système et les technologies retenues.

```mermaid
flowchart TD
    etudiant(["👤 Étudiant / Organisateur / Admin"])

    subgraph supevents["SupEvents [Système logiciel]"]
        frontend["🖥️ Frontend Web\n[React/Vite]\nSPA servie par CDN\nport 443"]
        gateway["🔀 API Gateway\n[nginx / Kong]\nRouting, SSL termination\nport 443"]
        backend["⚙️ API Backend\n[NestJS]\nLogique métier, REST API\nport 3000"]
        db["🗄️ PostgreSQL\n[Base de données]\nDonnées métier persistantes\nport 5432"]
        cache["⚡ Redis\n[Cache / Sessions]\nTokens, sessions, verrous\nport 6379"]
        broker["📨 RabbitMQ\n[Broker de messages]\nÉvénements asynchrones\nport 5672 AMQP"]
        worker["🔔 Notification Worker\n[Node.js]\nConsomme les événements\net envoie les emails"]
        storage["📦 Stockage objets\n[MinIO / S3]\nImages, médias, exports CSV\nport 9000"]
    end

    stripe["💳 Stripe"]
    sendgrid["📧 SendGrid"]
    sso["🔐 SSO École OIDC"]

    etudiant -- "HTTPS:443" --> frontend
    frontend -- "HTTPS:443\nAppels API REST" --> gateway
    gateway -- "HTTP:3000\nProxy vers backend" --> backend
    backend -- "TCP:5432\nLecture/écriture données" --> db
    backend -- "Redis:6379\nCache & verrous" --> cache
    backend -- "AMQP:5672\nPublie les événements" --> broker
    broker -- "AMQP:5672\nConsomme les événements" --> worker
    backend -- "HTTPS:443\nInitie paiement" --> stripe
    stripe -- "Webhook HTTPS\nConfirmation" --> gateway
    worker -- "HTTPS:443\nEnvoi email" --> sendgrid
    backend -- "OIDC:443\nValidation JWT" --> sso
    backend -- "HTTP:9000\nStockage fichiers" --> storage

    style frontend fill:#1168bd,color:#fff
    style gateway fill:#1168bd,color:#fff
    style backend fill:#1168bd,color:#fff
    style db fill:#1168bd,color:#fff
    style cache fill:#1168bd,color:#fff
    style broker fill:#1168bd,color:#fff
    style worker fill:#1168bd,color:#fff
    style storage fill:#1168bd,color:#fff
```

L'architecture en containers sépare clairement les préoccupations : le frontend React est une SPA statique servie par CDN, l'API Gateway centralise le routage et la terminaison SSL, le backend NestJS porte toute la logique métier. La communication asynchrone via RabbitMQ découple le service de notification du flux critique d'inscription — une panne du worker de notification n'empêche pas un étudiant de finaliser son inscription. Redis est utilisé à la fois pour le cache des sessions JWT et pour les verrous distribués sur la gestion des places.

## 6.1.3 Diagramme de composants — API Backend

Ce diagramme décompose l'intérieur du container API Backend en modules fonctionnels. Il est destiné aux développeurs qui implémentent les modules et qui doivent comprendre les dépendances internes.

```mermaid
flowchart LR
    gateway["API Gateway"]

    subgraph backend["API Backend [NestJS]"]
        auth["AuthModule\nIAuthService\nValidation JWT/OIDC\nGestion des rôles RBAC"]
        event["EventModule\nIEventService\nCycle de vie événement\nGestion jauge & catalogue"]
        ticket["TicketModule\nITicketService\nRéservation & confirmation\nGénération QR code"]
        payment["PaymentModule\nIPaymentService\nOrchestration Stripe\nIdempotence webhooks"]
        notif["NotificationModule\nINotificationService\nPublie sur RabbitMQ"]
        user["UserModule\nIUserService\nProfil & préférences\nDroit à l'oubli RGPD"]
    end

    db[("PostgreSQL")]
    cache[("Redis")]
    broker["RabbitMQ"]
    stripe["Stripe"]
    sso["SSO OIDC"]

    gateway --> auth
    gateway --> event
    gateway --> ticket
    gateway --> payment
    gateway --> user

    auth --> sso
    auth --> cache

    ticket --> event
    ticket --> payment
    ticket --> notif
    ticket --> db

    payment --> stripe
    payment --> db
    payment --> cache

    event --> db
    event --> storage["MinIO/S3"]

    user --> db

    notif --> broker
```

Les dépendances sont tracées selon les règles métier réelles : `TicketModule` dépend de `EventModule` (vérification des places disponibles) et de `PaymentModule` (initiation du paiement), mais pas directement de `NotificationModule` — le découplage est assuré via RabbitMQ. `AuthModule` est transverse et consulté par tous les modules pour la validation RBAC, mais n'est pas représenté comme dépendance directe de chaque module pour ne pas surcharger le diagramme.
