# §6.2 — Vue des processus

## 6.2.1 Diagramme de séquence : Inscription à un événement (parcours nominal)

Ce diagramme couvre le flux complet déclenché lorsqu'un étudiant clique sur "S'inscrire" jusqu'à la réception de son ticket par email. Il est destiné aux développeurs qui implémentent les modules `TicketModule` et `PaymentModule`, et sert de référence pour les tests d'intégration du parcours critique.

```mermaid
sequenceDiagram
    actor Étudiant
    participant FE as Frontend
    participant GW as API Gateway
    participant Auth as AuthModule
    participant Event as EventModule
    participant Ticket as TicketModule
    participant Payment as PaymentModule
    participant DB as PostgreSQL
    participant Stripe
    participant MQ as RabbitMQ
    participant Notif as NotificationWorker

    Étudiant->>FE: Clic "S'inscrire" (eventId)
    FE->>GW: POST /api/v1/tickets (JWT, eventId)
    GW->>Auth: Valider JWT
    Auth-->>GW: Identité validée (userId, role)

    GW->>Event: Vérifier disponibilité(eventId)
    Event->>DB: SELECT places_disponibles FOR UPDATE
    DB-->>Event: N places disponibles

    alt Places disponibles
        Event-->>GW: OK (placesRestantes)
        GW->>Ticket: Créer réservation pending(userId, eventId)
        Ticket->>DB: INSERT ticket {statut: "pending"}
        DB-->>Ticket: ticketId

        Ticket->>Payment: Initier paiement(ticketId, montant)
        Payment->>Stripe: POST /payment_intents
        Stripe-->>Payment: {paymentIntentId, clientSecret}
        Payment-->>FE: {clientSecret} (via GW)

        FE->>Étudiant: Afficher formulaire de paiement Stripe
        Étudiant->>Stripe: Saisir données carte
        Stripe-->>Étudiant: Confirmation paiement
        Stripe->>GW: Webhook POST /webhooks/stripe (payment_intent.succeeded)

        GW->>Payment: Traiter webhook Stripe(event)
        Payment->>DB: UPDATE ticket {statut: "confirmed"}, créer Payment
        Payment->>Ticket: Générer QR code(ticketId)
        Ticket->>DB: UPDATE ticket {qrCode: "..."}

        Payment->>MQ: Publish ticket.confirmed {ticketId, userId, email}
        MQ-->>Notif: Consomme ticket.confirmed
        Notif->>Étudiant: Email "Votre ticket est confirmé" (avec QR code)

        Payment-->>FE: 200 OK (ticketId, statut: "confirmed")
        FE-->>Étudiant: Page de confirmation avec QR code

    else Plus de places
        Event-->>GW: 409 Conflict (complet)

        opt Liste d'attente activée
            GW->>Ticket: Ajouter à la liste d'attente(userId, eventId)
            Ticket->>DB: INSERT waiting_list_entry
            Ticket->>MQ: Publish waiting_list.joined
            MQ-->>Notif: Consomme waiting_list.joined
            Notif->>Étudiant: Email "Vous êtes sur liste d'attente"
        end

        GW-->>FE: 409 Conflict
        FE-->>Étudiant: Message "Événement complet"
    end
```

Les points clés de ce flux : le verrou `SELECT FOR UPDATE` sur les places garantit l'absence de surbooking même en cas de requêtes concurrentes. La réservation est créée en statut `pending` avant le paiement — si le paiement échoue, le ticket est annulé et la place libérée (cf. §6.2.2). La notification est découplée via RabbitMQ : une indisponibilité de SendGrid ne bloque pas la confirmation du ticket.

## 6.2.2 Diagramme de séquence : Échec de paiement

Ce diagramme reprend le flux d'inscription jusqu'à l'étape de paiement Stripe, mais traite les cas d'erreur. Il est destiné aux développeurs du `PaymentModule` et aux testeurs qui doivent couvrir les scénarios d'échec dans les tests d'intégration.

```mermaid
sequenceDiagram
    actor Étudiant
    participant FE as Frontend
    participant GW as API Gateway
    participant Payment as PaymentModule
    participant Ticket as TicketModule
    participant DB as PostgreSQL
    participant Stripe
    participant MQ as RabbitMQ
    participant Notif as NotificationWorker
    participant Log as Logger

    Note over FE,DB: [Ticket en statut "pending" déjà créé — cf. §6.2.1]

    Étudiant->>Stripe: Saisir données carte
    Stripe->>GW: Webhook POST /webhooks/stripe

    alt Refus explicite (payment_intent.payment_failed)
        GW->>Payment: Traiter webhook échec(paymentIntentId, reason)
        Payment->>DB: UPDATE ticket {statut: "cancelled"}
        Payment->>DB: UPDATE event {places_disponibles + 1}
        Payment->>Log: Log échec paiement {ticketId, reason, timestamp}
        Payment->>MQ: Publish payment.failed {ticketId, userId, reason}
        MQ-->>Notif: Consomme payment.failed
        Notif->>Étudiant: Email "Votre paiement a échoué — raison : [reason]"
        Payment-->>FE: 200 OK (webhook traité)
        FE-->>Étudiant: Page "Paiement refusé — réessayer ?"

    else Timeout Stripe (absence de webhook dans les 15 min)
        Note over Payment: Job CRON de réconciliation (toutes les 15 min)
        Payment->>Stripe: GET /payment_intents/{id} (status check)
        Stripe-->>Payment: {status: "requires_payment_method"} ou "canceled"
        Payment->>DB: UPDATE ticket {statut: "cancelled"}
        Payment->>DB: UPDATE event {places_disponibles + 1}
        Payment->>Log: Log timeout paiement {ticketId, elapsed}
        Payment->>MQ: Publish payment.timed_out {ticketId, userId}
        MQ-->>Notif: Consomme payment.timed_out
        Notif->>Étudiant: Email "Votre réservation a expiré — réessayer ?"
    end
```

Les deux cas d'échec (refus explicite et timeout) mènent au même résultat : le ticket passe en `cancelled` et la place est restituée atomiquement. La distinction entre les deux flux est importante pour la notification : en cas de refus, la raison métier est communiquée à l'étudiant (carte refusée, fonds insuffisants) ; en cas de timeout, le message est plus générique. Le job CRON de réconciliation garantit qu'aucune réservation ne reste indéfiniment en `pending` même si le webhook Stripe est perdu.
